# RULE-040 Bug 生命周期管理

> 生效日期: 2026-06-14
> 适用范围: GraceOS 项目
> 维护者: 按项目运营规范自动执行

---

## 目录结构

```
BUG_REPORTS/
├── active/           ← 未解决 / 7天内新增
├── resolved/         ← 已修复，保留根因+修复方案
├── archive/          ← 超过30天 / 无复用价值
├── BUG_TEMPLATE.md   ← Bug 报告模板
└── README.md         ← 本文件（规则说明）
```

---

## 1. Active（活跃）

**准入条件**（满足任一）：
- Bug 未被修复
- 最近 7 天内创建

**文件名格式**: `BUG-{编号}-{简短描述}.md`

**必须包含**：
- 发现日期
- 触发条件（复现步骤）
- 影响范围（哪些表/功能/用户）
- 严重级别（P0/P1/P2/P3）
- 当前状态（未修复）

**示例**：
```
BUG_REPORTS/active/BUG-001-disks-table-missing.md
```

---

## 2. Resolved（已解决）

**准入条件**：
- Bug 已被修复并验证通过

**迁移源**：`active/` → `resolved/`

**必须包含**（在 active 基础上追加）：
- 根因分析
- 修复方案（具体代码/配置变更）
- 修复日期
- 修复 commit（如有）
- 验证方法（如何确认已修复）

**保留价值**：
- 防止同类问题重复出现
- 作为回归测试用例来源
- 新人培训参考

**示例**：
```
BUG_REPORTS/resolved/BUG-001-disks-table-missing.md
```

---

## 3. Archive（归档）

**准入条件**（满足任一）：
- resolved 状态超过 30 天
- 经评估无复用价值（如环境特定、已过时）

**迁移源**：`resolved/` → `archive/`

**保留内容**：
- 原始报告摘要
- 修复结论
- 归档日期

**目的**：
- 历史记录可追溯
- 不占用 active/resolved 视野

---

## 迁移规则

```
┌──────────┐    修复完成     ┌──────────┐   30天未使用    ┌──────────┐
│  active  │ ──────────────→ │ resolved │ ──────────────→ │ archive  │
└──────────┘                 └──────────┘                 └──────────┘
     ↑                                                        │
     └──────────────── 如再次出现 ←──────────────────────────┘
```

| 触发条件 | 操作 |
|----------|------|
| Bug 修复完成 + 验证通过 | active → resolved |
| 已在 resolved 且超过 30 天 | resolved → archive |
| 已归档 Bug 再次出现 | archive → active（从归档恢复，追加新触发条件） |

---

## 四项原则

1. **不删除已解决 Bug** — 已修复的 Bug 永远保留在 resolved 或 archive
2. **不丢失根因信息** — 每个 Bug 必须记录触发条件 + 根因 + 修复方案
3. **保证可追溯性** — 文件名编号不变，三个阶段之间编号连续
4. **保持轻量化** — 单个 Markdown 文件，不引入外部系统，不增加评审流程

---

## Bug 严重级别

| 级别 | 定义 | 示例 |
|------|------|------|
| **P0** | 阻断性故障，应用无法运行 | dashboard.py 崩溃、数据库无法连接 |
| **P1** | 核心功能不可用 | 文件搜索无结果、删除无法执行 |
| **P2** | 功能异常但有 workaround | UI 错位、数据显示不全 |
| **P3** | 体验问题 | 样式问题、文案错误 |
