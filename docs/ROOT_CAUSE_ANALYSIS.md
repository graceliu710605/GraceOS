## 根因分析

### 根因 1: 重复文件「大小全部显示0」

**不是 Bug，是显示格式问题。**

数据库 154,048 组重复文件中，**前 50 组按重复数量排序**，恰好全部是极小文件（LOCK、__init__.py 等）。

- `ROUND(file_size/1048576.0,2) AS size_mb` 将 < 10KB 的文件四舍五入为 0.00 MB
- 显示 `f"{row['size_mb']:.1f}"` → 显示 0.0
- 有 > 1MB 的重复文件（如 data_2 1.0MB×71、data_3 4.0MB×64），但排在后面，未在前 50 组显示

**影响**: 用户看到前 50 组全是 0.0，误以为所有重复文件大小为 0。

### 根因 2: 文件名点击无法打开 / 删除 WinError 2

**数据库记录与实际文件系统不同步。**

扫描器 (`file_scanner.py`) 首次运行后生成 115 万条记录存入 SQLite。此后文件系统持续变化：

- Chrome 自动更新：删除旧版本目录（如 `148.0.7778.217`）
- 用户手动删除/移动文件
- 临时文件/缓存被系统自动清理
- `pagefile.sys` 等系统文件无法直接打开

验证数据：在 50,000 条记录中已发现 5 个文件不存在（均为 Chrome 旧版本遗留）。

Top 10 大文件中，5/10 已不存在于磁盘（pagefile.sys、多个 rar 文件）。

### 根因 3: Top100 大文件异常

同上根因 2。大文件列表中 50% 的记录指向不存在的文件。点击打开/删除时触发 WinError 2。

## 修复方案（统一）

不是逐个页面打补丁。三个根因对应三个修复：

### 修复 A: 重复文件大小显示
- 显示格式从纯 MB 改为自适应：< 1MB 显示 KB，< 1KB 显示 bytes
- 或在前 50 组中穿插大文件组

### 修复 B: 数据库与实际文件系统同步
- 所有模块在渲染前增加 `os.path.exists()` 检查
- 不存在的文件标记「文件已删除」并跳过操作
- `_safe_delete` 增加文件存在性检查，不存在时静默跳过而非报错

### 修复 C: 删除/打开操作增强
- `_open_file` 已处理不存在（toast），但需显示提示信息时包含路径
- `_safe_delete` 在 `shutil.move` 前增加 `os.path.exists()` 预检查，避免 WinError 2
